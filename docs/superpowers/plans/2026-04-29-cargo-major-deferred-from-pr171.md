# Cargo major-version upgrades deferred from #171

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Land the seven major-version bumps Dependabot proposed in #171
(closed without merging) as three smaller, independently-reviewable PRs.
Five crates upgrade with zero source-code changes; one (`fs4`) needs a
small migration in `rimap-audit`; one (`rand`) is deferred until upstream
consumers (`governor`, `proptest`, `ulid`, `chrono`) catch up.

**Architecture:** No design changes. The workspace already declares
every affected dependency once in the root `Cargo.toml`'s
`[workspace.dependencies]`; member crates inherit via `foo = { workspace =
true }`. The work is dependency-version edits plus, for `fs4`, three
import-path updates inside `crates/rimap-audit`.

**Tech Stack:** Cargo workspace, `cargo update`, `cargo deny`, the existing
CI gates (rustfmt, clippy, test stable, test MSRV 1.88.0, cargo-deny,
zizmor, SonarQube).

---

## Background

PR #171 grouped seven major bumps and failed CI with three concrete blockers:

1. `fs4 0.13 → 1.x` removed the `fs4::fs_std::FileExt` module path
   (three `unresolved import` errors in `rimap-audit`).
2. `rand 0.9 → 0.10` reorganised `RngCore` and removed
   `ThreadRng::fill_bytes` (one `E0599` plus one `unresolved import` in
   `rimap-audit`).
3. `cargo-deny`'s `[bans] multiple-versions = "deny"` rule trips on the
   resulting `rand 0.9.4 + rand 0.10.1` duplicate, because `governor`,
   `proptest`, `ulid`, and `chrono` still pin `rand 0.9`.

The other five crates in the group (`toml`, `directories`, `sha2`,
`strum`, `linkify`) compiled cleanly. Splitting them out lets us land
99 % of the upgrade churn now without waiting on `rand`'s ecosystem.

## Strategy: three PRs

| PR | Bumps | Source changes | Risk | Status |
|----|-------|----------------|------|--------|
| **A** | `toml 0.8→1`, `directories 5→6`, `sha2 0.10→0.11`, `strum 0.26→0.28`, `linkify 0.10→0.11` | None expected | Low | Land first |
| **B** | `fs4 0.13→1.x` | Three import-path updates in `rimap-audit` | Medium (file-locking is a security primitive) | Land second |
| **C** | `rand 0.9→0.10` | RNG call-site rewrite in `rimap-audit` + `[bans] skip` for `rand 0.9` until upstream catches up | High (audit-salt path; introduces an accepted duplicate) | **Deferred** — track and revisit |

PR A and PR B can land in either order; PR B touches `rimap-audit`
exclusively, PR A touches none of the same files. PR C waits on
upstream movement (see "Deferred work" below).

## Task 2: PR B — `fs4` 0.13 → 1.x

**Files:** `Cargo.toml`, `Cargo.lock`, three files under
`crates/rimap-audit/src/`, possibly `deny.toml`.

- [ ] **Step 1: Branch from updated main**

```bash
git checkout main && git pull --ff-only origin main
git checkout -b deps/fs4-1x
```

If PR A has merged, the branch starts from a tree where the safe-five
are already on the new versions. If not, this PR can still land
independently — the two are non-overlapping.

- [ ] **Step 2: Read the fs4 1.x API surface BEFORE editing code**

The 0.13 → 1.x change reorganised module paths. Confirm the current
shape on docs.rs and in `cargo doc --open`:

```bash
cargo add --dry-run fs4@1
# Then on docs.rs:
#   https://docs.rs/fs4/latest/fs4/
#   https://docs.rs/fs4/latest/fs4/trait.FileExt.html
```

Specifically determine:

1. Where the `FileExt` trait now lives. PR #171's failure logs show
   `fs4::fs_std` is gone in 1.x. The most likely new path is
   `fs4::FileExt` at the crate root (the docs page lists `FileExt` in
   the top-level traits list); verify by reading the trait page.
2. Whether `try_lock_exclusive(&File) -> io::Result<bool>` and
   `try_lock_shared(&File) -> io::Result<bool>` retained their
   signatures, or whether they were renamed/refactored to return
   `Result<(), TryLockError>` (note: fs4 1.x added a `TryLockError`
   enum — visible in the index — which suggests the try-lock API may
   now use `Result<()>` with a typed error rather than `Result<bool>`).
3. Whether the implicit-on-drop release is still the contract, or
   whether 1.x requires an explicit `unlock()` call.

The audit writer's lock invariant ("one `AuditWriter` holds `LOCK_EX`
for its entire lifetime; the lock is released implicitly on drop")
is a security primitive. If 1.x changes the drop semantics, the
migration MUST preserve the lifetime contract — file an ADR or update
`crates/rimap-audit/src/writer/mod.rs`'s module-doc comment to match
whatever 1.x actually guarantees, and add a Drop impl if the previous
implicit cleanup no longer fires.

- [ ] **Step 3: Edit `Cargo.toml`**

```toml
fs4 = { version = "1", default-features = false, features = ["sync"] }
```

(The feature set may also have shifted between 0.13 and 1.x. Confirm
that `sync` is still the right gate to pin to the std-fs blocking
implementation rather than tokio/async-std/smol; consult the
`features` table from `cargo info fs4`.)

- [ ] **Step 4: Update the three call sites**

Files (per `rg -n 'fs4::|fs_std' crates/rimap-audit/src/`):

- `crates/rimap-audit/src/writer/mod.rs:20` — `use fs4::fs_std::FileExt;`
- `crates/rimap-audit/src/writer/rotation.rs:7` — same.
- `crates/rimap-audit/src/reader/mod.rs:7` — same.

Replace each `use fs4::fs_std::FileExt;` with the path determined in
Step 2 (likely `use fs4::FileExt;`). Then walk every call site:

```bash
rg -n 'FileExt::try_lock_exclusive|FileExt::try_lock_shared' crates/rimap-audit/src/
```

For each call:

- If 1.x kept `Result<bool>`: no logic change; just the import.
- If 1.x switched to `Result<(), TryLockError>`: replace
  `match FileExt::try_lock_exclusive(&file) { Ok(true) => …, Ok(false) => …, Err(e) => … }`
  with `match FileExt::try_lock_exclusive(&file) { Ok(()) => …, Err(TryLockError::WouldBlock) => …, Err(TryLockError::Io(e)) => … }`.
  The `AuditError::Locked` semantic must still surface for the
  "another writer holds the lock" branch — preserve that mapping.

Update the doc comment at `writer/mod.rs:1-12` to reference the new
import path (the comment mentions `try_lock_exclusive` by name).

- [ ] **Step 5: Verify lock-drop semantics with a real test**

The existing `crates/rimap-audit/tests/concurrent_lock.rs`
(and any `writer` test that opens two `AuditWriter`s against the
same path) is the regression test for the lock contract. Run it
specifically:

```bash
cargo nextest run -p rimap-audit concurrent_lock
```

Expected: the second `AuditWriter::open` fails with
`AuditError::Locked`, exactly as it did under fs4 0.13. If this test
flakes or passes incorrectly under the new API, stop and re-read
Step 2 before adjusting the test.

If the 1.x API requires an explicit `unlock()` call before drop to
release the OS-level lock, add a `Drop` impl on `AuditWriter` that
calls `FileExt::unlock(&self.file)` and document the change in the
module comment.

- [ ] **Step 6: Re-check `deny.toml` skip-tree**

PR #171's cargo-deny run produced two warnings:

```

  ./deny.toml:110:15  windows-sys 0.48
  ./deny.toml:111:15  windows-sys 0.52

`fs4 1.x` consolidates onto `windows-sys 0.61`. After the bump, run:

If `windows-sys 0.48` and/or `0.52` no longer appear, delete the
matching entries from `deny.toml`'s `[bans] skip-tree`. Leave `0.59`
unless it's also gone. Update the surrounding block-comment if the
"each transitive consumer pins a different release" inventory has
changed.

- [ ] **Step 7: Full local CI**

```

Expected: every gate green. Pay special attention to:

- `cargo nextest run -p rimap-audit` — every audit test must pass; the

## Deferred work: `rand` 0.9 → 0.10 (PR C, not yet ready)

The `rand` 0.10 bump is gated on FOUR upstream consumers releasing a
`rand 0.10`-compatible version:

- `governor` (latest 0.10.4, still on `rand 0.9`)
- `proptest` (latest 1.11.0, still on `rand 0.9`)
- `ulid` (latest 1.2.1, still on `rand 0.9`)
- `chrono` (latest, still on `rand 0.9` via the `rand` feature)

Until at least three of those bump, attempting `rand 0.10` produces a
duplicate that `cargo deny`'s `[bans]` rule rejects. Two paths:

1. **Wait.** Re-check the four crates monthly with
   `cargo info governor proptest ulid chrono | rg rand`. When two-or-more
   move, revisit.
2. **Accept the duplicate now** by adding `{ name = "rand", version =
   "0.9" }` to `deny.toml`'s `[bans] skip` (mirroring the existing
   `getrandom` 0.2/0.3/0.4 stanza) AND doing the source-side migration:

   - `crates/rimap-audit/src/redact/mod.rs:18` — replace
     `use rand::{RngCore, rng};` with the 0.10 trait path. The 0.10
     re-export is `rand::rand_core::RngCore` (verify on docs.rs); the
     `rng()` factory may have been renamed to `thread_rng()` or moved.
   - `crates/rimap-audit/src/redact/mod.rs:79` — replace
     `rng().fill_bytes(&mut bytes);`. In 0.10, `fill_bytes` was
     replaced; the new idiom is likely `rng().fill(&mut bytes[..])`
     against the `Fill` trait, OR `rand_core::RngCore::fill_bytes` if
     the trait is imported. Confirm against
     `https://docs.rs/rand/0.10.1/rand/` before writing the call.

   The redaction salt must remain a CSPRNG-quality 32 bytes — verify
   the new call still draws from the OS entropy pool (i.e. `ThreadRng`
   in 0.10 is still seeded from `OsRng`).

Choose path (1) until at least one of {governor, proptest} bumps to
`rand 0.10`. Until then, the PR-C work item lives only in this section
and need not be tracked separately.

## Out of scope

- **Bumping `rand 0.9 → 0.10` now.** Covered above. Re-evaluate when
  upstream catches up.
- **Coordinated workspace MSRV bump.** None of PR A or PR B should
  require it. If `cargo build` against MSRV 1.88.0 fails, treat as
  blocking and resolve in a separate ADR.
- **Replacing `governor` to escape the `rand 0.9` pin.** `governor`
  is the only viable rate-limiter on the workspace's feature surface;
  swapping it is a much larger decision that belongs to a separate
  spec, not a dependency-bump plan.
- **Cleanup of unrelated `[bans] skip` entries.** Only the `windows-sys
  0.48` / `0.52` entries are touched in PR B, and only if `cargo tree`
  confirms they no longer match. Leave the rest alone.
