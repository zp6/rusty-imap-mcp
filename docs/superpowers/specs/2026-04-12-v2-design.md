# Rusty IMAP MCP v2 — Design Specification

**Status:** Approved 2026-04-12
**Target:** v2.0.0
**Scope:** SMTP sending, message deletion, folder management, and a new `destructive`
security posture. Extends the v1 security-first architecture with irreversible
operations gated by posture, folder allowlists, and audit-before-action.

## 1. Goals & Non-Goals

### Goals

- Add six new MCP tools: `send_email`, `delete_message`, `expunge`, `create_folder`,
  `rename_folder`, `delete_folder`.
- Introduce a `destructive` posture above `full` for irreversible operations
  (`expunge`, `delete_folder`).
- Add SMTP support via a new `rimap-smtp` crate using `lettre`, with explicit
  `[smtp]` configuration — no inference from IMAP settings.
- Maintain defense-in-depth: posture gates, folder allowlists, protected folder
  lists, rate limiting, and audit-before-action for all destructive operations.
- Preserve full backward compatibility — existing v1 configs work unchanged.

### Non-Goals (v2)

- Bounce/DSN monitoring (v2 logs SMTP transaction results only; bounce parsing
  deferred to v2.x with IDLE support).
- Label management API (`add_label`/`remove_label`/`list_labels` — deferred to
  v2.x or v3 where provider abstraction is more important).
- SMTP setup wizard / onboarding script (useful but not blocking; deferred to v2.x).
- Connection pooling for SMTP (one-shot per send is sufficient at current scale).
- Retry logic for failed sends (fail fast; the agent decides whether to retry).
- Splitting `expunge_folders` and `delete_folder` allowlists (single list for now;
  splitting is a future refinement if usage patterns warrant it).

### Threat Model Additions

v2 introduces two new threat vectors beyond the v1 threat model:

- **Autonomous send risk:** A prompt-injected agent can now send email directly
  (at `full` posture), not just create drafts. Mitigations: rate limiting
  (`sends_per_minute`), audit logging with recipient hashes, Sent folder copy for
  human review.
- **Permanent data destruction:** `expunge` and `delete_folder` permanently remove
  messages. Mitigations: `destructive` posture gate, folder allowlist (default
  deny-all), protected folders list, audit-before-action with UID/count records.

The v1 threat model (crafted email → prompt injection → harmful action) remains
the primary adversary. v2 increases the blast radius of a successful injection,
which is why destructive operations require explicit opt-in at every layer.

## 2. Architecture Changes

### New Crate: `rimap-smtp`

A new workspace member alongside the existing seven crates:

```
crates/
├── rimap-core/       # shared types (Posture, ToolName, errors)
├── rimap-config/     # config loading, validation, credentials
├── rimap-imap/       # async-imap wrapper
├── rimap-content/    # MIME parsing, sanitization
├── rimap-audit/      # JSONL audit log
├── rimap-authz/      # posture matrix, rate limiter, circuit breaker
├── rimap-smtp/       # NEW — lettre wrapper, SMTP connection, TLS
└── rimap-server/     # MCP server, tool dispatch, main.rs
```

`rimap-smtp` is a thin wrapper around `lettre` with connection management,
TLS configuration (via `rustls`, same as the IMAP side), and error mapping.
It does not construct messages — message building is shared between
`create_draft` and `send_email` in `rimap-server`.

### Dependency Changes

- `rimap-server` gains a dependency on `rimap-smtp`.
- `lettre` added to workspace dependencies with `rustls-tls` feature (no
  `native-tls`).
- `rimap-smtp` depends on `rimap-core` (error types, config types) and `lettre`.

### Changes to Existing Crates

| Crate | Changes |
|-------|---------|
| `rimap-core` | Add 6 `ToolName` variants, `Posture::Destructive`, `ProtectedFolder` / `ExpungeDenied` errors |
| `rimap-config` | Add `[smtp]` section parsing, `protected_folders`, `expunge_folders`, `sends_per_minute`; remove `V2_TOOL_NAMES`; add `SmtpRequired` / `ConflictingFolders` validation |
| `rimap-imap` | Add `delete_message`, `expunge`, `create_folder`, `rename_folder`, `delete_folder` ops |
| `rimap-authz` | Expand posture matrix to 19 tools × 4 postures; add folder allowlist and protected folder checks |
| `rimap-audit` | No schema changes — existing JSONL format accommodates new tools |
| `rimap-server` | Add 6 tool handlers; extract shared message-building logic from `create_draft` |

## 3. New Tool Surface

### Tool Definitions

| Tool | Parameters | Behavior |
|------|-----------|----------|
| `send_email` | `to`, `cc` (opt), `bcc` (opt), `subject`, `body`, `in_reply_to` (opt), `references` (opt) | Build RFC 5322 message, SMTP send, APPEND copy to Sent folder. Returns Message-ID. |
| `delete_message` | `folder`, `uid` | STORE `+FLAGS (\Deleted)` then UID MOVE to Trash. If already in Trash, flag only. Returns confirmation. |
| `expunge` | `folder` | EXPUNGE on folder. Folder must be in `expunge_folders` allowlist. Removes `\Deleted`-flagged messages. Returns expunged count. |
| `create_folder` | `name` | IMAP CREATE. Returns confirmation. |
| `rename_folder` | `old_name`, `new_name` | IMAP RENAME. `old_name` must not be in `protected_folders`. Returns confirmation. |
| `delete_folder` | `name` | IMAP DELETE. Folder must be in `expunge_folders` allowlist and not in `protected_folders`. Returns confirmation. |

### Tool Handler Flows

**`send_email`:**

1. Authz check — posture `full` or `destructive`
2. Rate limit check — `sends_per_minute` (default 3)
3. Input validation — recipient count, subject length, body size (same bounds
   as `create_draft`)
4. Build RFC 5322 message via `mail-builder` (shared logic with `create_draft`)
5. SMTP send via `rimap-smtp`
6. APPEND copy to Sent folder via IMAP with `\Seen` flag. If APPEND fails,
   log warning but do not fail the send — email was already delivered.
7. Audit log — Message-ID, recipients (salted hash), SMTP response, Sent copy UID
8. Return Message-ID

**`delete_message`:**

1. Authz check — posture `full` or `destructive`
2. Validate folder and UID exist (EXAMINE + UID FETCH)
3. If in Trash: STORE `+FLAGS (\Deleted)`, done
4. If in other folder: STORE `+FLAGS (\Deleted)` then UID MOVE to Trash
5. Audit log — source folder, UID, Message-ID, destination (Trash)
6. Return confirmation with new UID in Trash

**`expunge`:**

1. Authz check — posture `destructive`
2. Folder allowlist check — `expunge_folders` (default empty = deny all)
3. EXAMINE folder, UID SEARCH DELETED for pre-deletion audit record
4. SELECT folder (read-write), issue EXPUNGE
5. Audit log — folder, expunged count, expunged UIDs
6. Return expunged count

**`create_folder`:**

1. Authz check — posture `full` or `destructive`
2. Input validation — name length (max 255 bytes), no null bytes, no path
   traversal, reject names colliding with `protected_folders` entries
   (case-insensitive)
3. IMAP CREATE
4. Audit log — folder name
5. Return confirmation

**`rename_folder`:**

1. Authz check — posture `full` or `destructive`
2. Protected folders check — reject if `old_name` in `protected_folders`
3. Input validation on `new_name` — same rules as `create_folder`
4. IMAP RENAME
5. Audit log — old name, new name
6. Return confirmation

**`delete_folder`:**

1. Authz check — posture `destructive`
2. Protected folders check — reject if `name` in `protected_folders`
3. Folder allowlist check — `expunge_folders` (same list as `expunge`)
4. EXAMINE folder, count messages for audit record
5. IMAP DELETE
6. Audit log — folder name, message count at deletion
7. Return confirmation

## 4. Posture Changes

### Posture Hierarchy

```
readonly < draft-safe < full < destructive
```

### Posture Matrix (19 tools × 4 postures)

| Tool | readonly | draft-safe | full | destructive |
|------|----------|------------|------|-------------|
| `list_folders` | yes | yes | yes | yes |
| `search` | yes | yes | yes | yes |
| `search_advanced` | - | - | yes | yes |
| `fetch_message` | yes | yes | yes | yes |
| `fetch_message_html` | - | - | yes | yes |
| `list_attachments` | yes | yes | yes | yes |
| `download_attachment` | yes | yes | yes | yes |
| `mark_read` | - | yes | yes | yes |
| `mark_unread` | - | yes | yes | yes |
| `flag` | - | yes | yes | yes |
| `unflag` | - | yes | yes | yes |
| `move_message` | - | yes | yes | yes |
| `create_draft` | - | yes | yes | yes |
| `send_email` | - | - | yes | yes |
| `delete_message` | - | - | yes | yes |
| `create_folder` | - | - | yes | yes |
| `rename_folder` | - | - | yes | yes |
| `expunge` | - | - | - | yes |
| `delete_folder` | - | - | - | yes |

The `POSTURE_MATRIX` const in `rimap-authz` expands from `[bool; 3]` to
`[bool; 4]`. Per-tool overrides (allow/deny) continue to work for all tools.

### `V2_TOOL_NAMES` Removal

The `V2_TOOL_NAMES` constant and `ParseToolNameError::V2` variant in `rimap-core`
are removed. The six tool names become real `ToolName` enum variants. Existing
v1 configs that do not reference these tools continue to work unchanged.

## 5. SMTP Architecture

### `rimap-smtp` Crate

**Responsibilities:**

- `SmtpClient` — wraps `lettre::AsyncSmtpTransport`, holds TLS config
- `send()` — takes a built `mail_builder::Message`, sends via lettre, returns
  SMTP response code and Message-ID
- Error types — `SmtpError` with variants: `Connection`, `Auth`, `Tls`,
  `Rejected`, `Timeout`

**What it does NOT do:**

- Message construction (stays in `rimap-server`)
- Bounce monitoring (log SMTP response only)
- Connection pooling (one-shot per send)
- Retry logic (fail fast)

### Connection Lifecycle

One SMTP connection per `send_email` call. SMTP is stateless per transaction,
unlike IMAP's persistent session. Connection flow:

1. Open TCP connection to configured `host:port`
2. TLS handshake — STARTTLS (port 587) or implicit TLS (port 465), via `rustls`
3. SMTP AUTH with configured credentials
4. MAIL FROM / RCPT TO / DATA
5. Read server response (250 OK or error)
6. QUIT and close

### TLS Configuration

Same approach as `rimap-imap`:
- `rustls` only (no `native-tls`)
- System root certificates via `webpki-roots`
- Connection timeout and command timeout configurable

### Shared Message Building

The RFC 5322 message construction logic currently in the `create_draft` handler
is extracted into a shared function. Both `create_draft` and `send_email` call it:

- `create_draft` → shared builder → IMAP APPEND to Drafts with `$PendingReview`
- `send_email` → shared builder → SMTP send → IMAP APPEND to Sent with `\Seen`

### Sent Folder Copy

After a successful SMTP send, APPEND the message to the Sent folder via IMAP
with `\Seen` flag. This matches standard mail client behavior. If the APPEND
fails, log a warning in the audit log but do not fail the `send_email` call —
the email was already delivered. The Sent folder name is not configurable in v2
(hardcoded to `"Sent"`); provider-specific Sent folder names are a v3
multi-account concern.

## 6. Configuration Changes

### New Config Sections

```toml
[smtp]
host = "smtp.example.com"
port = 587                          # 587 for STARTTLS, 465 for implicit TLS
encryption = "starttls"             # "starttls" | "tls" | "none"
username = "user@example.com"
credential = "keyring:smtp"         # same resolution as IMAP credentials
```

### New Fields in Existing Sections

```toml
[security]
posture = "full"                    # new value: "destructive"
protected_folders = [               # default value shown
    "INBOX", "Sent", "Drafts", "Trash"
]
expunge_folders = []                # default empty — deny all expunge/delete_folder

[limits]
sends_per_minute = 3                # new, default 3
```

### Validation Rules

- **`SmtpRequired`:** If `send_email` is effectively enabled (posture + per-tool
  overrides resolve to allow) but no `[smtp]` section exists, startup error.
  If the user explicitly denies `send_email` via override, the `[smtp]` section
  is not required.
- **`ConflictingFolders`:** If a folder appears in both `protected_folders` and
  `expunge_folders`, startup error. These lists are contradictory — a folder
  cannot be both protected from deletion and eligible for expunge.
- **`protected_folders` matching:** Case-insensitive (INBOX = inbox = Inbox).
- **Credential resolution:** `[smtp].credential` uses the same backends as IMAP:
  `keyring:<service>`, `env:<VAR>`, or literal string. No new backends.

### Backward Compatibility

Existing v1 configs work unchanged:
- No `[smtp]` section → `send_email` not available
- `posture = "draft-safe"` (default) → no v2 tools enabled
- `protected_folders` defaults to well-known folders if not specified
- `expunge_folders` defaults to empty if not specified

## 7. Folder Safety

### Protected Folders

The `protected_folders` config option prevents `delete_folder` and
`rename_folder` from operating on critical mailboxes.

**Default:** `["INBOX", "Sent", "Drafts", "Trash"]`

**Behavior:**
- `delete_folder(name)` where `name` matches a protected folder → `ProtectedFolder` error
- `rename_folder(old_name, ...)` where `old_name` matches a protected folder → `ProtectedFolder` error
- Matching is case-insensitive
- Users can extend the list (e.g., add `"Archives"`, `"Legal Hold"`)
- Users can narrow the list by removing entries (explicit opt-out)

**INBOX hard block:** `delete_folder("INBOX")` and `rename_folder("INBOX", ...)`
are always rejected regardless of `protected_folders` configuration. This is a
hardcoded check, not configurable. RFC 3501 prohibits deleting INBOX, but server
implementations vary — the client-side block ensures consistent behavior.

### Expunge Folder Allowlist

The `expunge_folders` config option controls which folders allow permanent
message removal.

**Default:** `[]` (empty — deny all)

**Applies to:**
- `expunge(folder)` — folder must be in the list
- `delete_folder(name)` — folder must be in the list (reuses same allowlist)

**Typical configuration:** `expunge_folders = ["Trash"]` enables the two-step
delete-then-expunge flow: `delete_message` moves to Trash (recoverable),
then `expunge("Trash")` permanently removes (irreversible, requires
`destructive` posture).

**Future refinement:** Splitting into separate `expunge_folders` and
`deletable_folders` lists if usage patterns show the shared list is too
coarse.

## 8. Audit Log Changes

### New Audit Event Types

| Event | Key Fields |
|-------|-----------|
| `send_email` | `message_id`, `recipients` (salted hash), `smtp_response`, `sent_copy_uid`, `folder` (Sent) |
| `delete_message` | `source_folder`, `uid`, `message_id`, `destination` (Trash) |
| `expunge` | `folder`, `expunged_count`, `expunged_uids` |
| `create_folder` | `folder` |
| `rename_folder` | `old_name`, `new_name` |
| `delete_folder` | `folder`, `message_count` |

### Redaction Rules

- **Recipient addresses** in `send_email`: salted hash, consistent with existing
  search criteria redaction.
- **Subject lines:** not logged (same policy as v1 `create_draft`).
- **Folder names:** logged verbatim (not PII; needed for forensic reconstruction).
- **Message-IDs:** logged verbatim (needed for send/bounce correlation).

### Pre-Deletion Audit

`expunge` and `delete_folder` log UIDs and message counts *before* executing
the destructive IMAP command. If the IMAP command succeeds but the post-action
audit write fails, the pre-deletion record still exists. This follows the v1
audit-before-action pattern.

### Schema

No JSONL schema changes. The existing `tool`, `params`, `result` fields
accommodate new tools without migration. New event types use the same
envelope format as v1 tools.

## 9. Error Handling

### New Error Types

| Error | Crate | Trigger |
|-------|-------|---------|
| `SmtpError::Connection` | `rimap-smtp` | Cannot reach SMTP server |
| `SmtpError::Auth` | `rimap-smtp` | SMTP authentication failed |
| `SmtpError::Tls` | `rimap-smtp` | TLS handshake failure |
| `SmtpError::Rejected` | `rimap-smtp` | Server rejected message (4xx/5xx) |
| `SmtpError::Timeout` | `rimap-smtp` | SMTP command timeout |
| `Error::ProtectedFolder` | `rimap-core` | Operation on a protected folder |
| `Error::ExpungeDenied` | `rimap-core` | Folder not in `expunge_folders` allowlist |
| `ConfigError::SmtpRequired` | `rimap-config` | `send_email` enabled but no `[smtp]` section |
| `ConfigError::ConflictingFolders` | `rimap-config` | Folder in both `protected_folders` and `expunge_folders` |

### Error Principles

- **Actionable messages:** `ExpungeDenied` names the rejected folder and suggests
  adding it to `expunge_folders`. `SmtpRequired` names the missing config section.
- **No information disclosure:** SMTP server banners and detailed rejection
  reasons are logged to audit but not returned to the MCP client. Tool responses
  get a generic error category, not raw server output.
- **Existing `AuthzError::PostureDenied`** covers the new `destructive` posture
  without new variants.

## 10. Testing Strategy

### Unit Tests (per crate)

| Crate | Coverage |
|-------|----------|
| `rimap-core` | New `ToolName` variants parse correctly; `Posture::Destructive` ordering; `protected_folders` case-insensitive matching |
| `rimap-config` | `[smtp]` section parsing; `SmtpRequired` fires when `send_email` enabled without `[smtp]`; `SmtpRequired` does not fire when `send_email` explicitly denied; `ConflictingFolders` detection; v2 tool names now parse as valid `ToolName` variants |
| `rimap-authz` | Expanded posture matrix (19 tools × 4 postures); `expunge_folders` allowlist acceptance/rejection; `ProtectedFolder` rejection for delete/rename; INBOX hard block |
| `rimap-smtp` | `SmtpClient` construction from config; error mapping from `lettre` errors; TLS config building |
| `rimap-server` | Tool handler input validation; shared message-building extraction; folder name validation (length, null bytes, path traversal, well-known collision) |

### Integration Tests (`rimap-imap`)

Extend the existing Podman-based Dovecot test harness:

- `delete_message` — flag + move to Trash, verify UID in Trash
- `expunge` — delete, expunge, verify message gone via UID SEARCH
- `create_folder` / `rename_folder` / `delete_folder` — round-trip against
  real Dovecot server
- Protected folder rejection — attempt delete/rename on INBOX, verify
  client-side rejection before any IMAP command is issued

### SMTP Integration Tests

Spin up a test SMTP server (MailHog or smtp4dev) in Podman alongside Dovecot.
The test SMTP server is a **sink** — it accepts messages, stores them in memory
for API inspection, and never attempts delivery. No DNS lookups, no MX
resolution, no outbound connections.

- `send_email` — send message, verify arrival via test server's API
- Sent folder copy — verify APPEND to Sent after successful send
- Auth failure — bad credentials, verify `SmtpError::Auth`
- Rate limiting — exceed `sends_per_minute`, verify rejection
- SMTP unavailable — verify `SmtpError::Connection` and audit log entry

Same patterns as existing tests: `cargo test`, Podman compose,
`rimap-it-*` container namespace, ephemeral teardown.

## 11. Post-v2 Roadmap

Each item gets its own spec when taken up. Listed for context, not commitment.

- **v2.x — Label management:** `add_label` / `remove_label` / `list_labels`
  tools abstracting over provider differences (Gmail folders vs IMAP keywords
  vs Exchange categories). Natural fit alongside v3 multi-account work.
- **v2.x — SMTP setup wizard:** Interactive onboarding script for SMTP
  credential configuration. Separate from the server binary.
- **v2.x — Allowlist splitting:** Separate `expunge_folders` and
  `deletable_folders` config if usage patterns warrant finer-grained control.
- **v2.x — IDLE & push:** Background IDLE, in-memory message index, MCP
  `notifications/resources/updated` (unchanged from v1 roadmap).
- **v3 — Multi-account:** `account` parameter on every tool, per-account
  audit logs, postures, and SMTP configs. Sent folder name becomes
  per-account configurable.
- **v3.x — HTTP transport:** Streamable HTTP per MCP spec, token-based auth.
- **v4 — OAuth2 / XOAUTH2:** Gmail and O365 compatibility.
